Module 5

Scénario pratique autonome

J1 · 16h15—17h00 · 45 min · Mise en situation

Les apprenants travaillent en autonomie sur un lab réaliste contenant plusieurs vulnérabilités à enchaîner. Le formateur circule pour débloquer et guider. Objectif : compromission complète d'un compte administrateur et exfiltration de données. Chaque étape doit être documentée dans une matrice ATT&CK personnalisée.

1 Briefing de mission

Contexte

Vous êtes Red Team pour le compte du groupe EcoVault, une startup fintech spécialisée dans le paiement en ligne. Après une suspicion de fuite de données, le RSSI vous mandate pour un pentest boîte grise sur leur nouvelle application web.

Vous disposez d'un compte utilisateur standard : user@ecovault.com / User2026!

Objectifs (par difficulté croissante)

  1. 🎯 Flag 1 — Trouver et exploiter une faille d'IDOR pour lire le profil d'un autre utilisateur
  2. 🎯 Flag 2 — Exploiter une injection SQL pour extraire la table users
  3. 🎯 Flag 3 — Contourner le mécanisme d'authentification pour devenir administrateur
  4. 🎯 Flag 4 — Exploiter une SSTI dans le panneau admin pour obtenir une RCE
  5. 🎯 Flag 5 (Bonus) — Pivoter vers le réseau interne et récupérer un fichier sensible sur un serveur SMTP

2 Règles de l'exercice

  • ✅ ROE autorisée : tout le lab (10.0.0.0/24), services HTTP, DB, SMTP
  • ❌ Interdit : DoS, destruction de données, sortir du scope réseau
  • 📝 Documentation obligatoire : chaque vulnérabilité trouvée doit être taggée avec son TXXXX MITRE correspondant
  • ⏰ Temps : 45 minutes. Les flags 1-3 sont obligatoires, 4-5 bonus.
  • 💡 Indices : disponibles, mais coûtent du temps (chaque indice = -5 min). Cliquer sur "Afficher l'indice" ci-dessous.

Conformité NIS2 & Rapport de pentest

Le rapport que vous documenterez avec les tags ATT&CK constitue un livrable de conformité NIS2. L'article 21 exige des « politiques de gestion des risques pour la sécurité des systèmes d'information » — votre heat map ATT&CK en est la matérialisation technique. Pensez à cartographier chaque vulnérabilité vers une recommandation actionnable.

3 Cibles

Environnement de lab

# Application web principale
URL:    http://ecovault.local
Ports:  80 (HTTP), 443 (HTTPS)

# Compte fourni
Login:  user@ecovault.com
Pass:   User2026!

# Réseau interne (accessible après compromission initiale)
Subnet: 10.0.0.0/24
SMTP:   10.0.0.10:25

4 Indices (à utiliser avec parcimonie)

Regarde l'endpoint /api/profile?id=XX. Que se passe-t-il si tu changes l'ID ? Utilise Burp Intruder pour énumérer les IDs de 1 à 100.
Le paramètre filter dans /api/transactions?filter= semble vulnérable à une injection SQL. Teste avec filter=' OR '1'='1. Pour l'extraction, utilise sqlmap ou un script time-based.
Analyse le JWT après connexion (jwt.io). Le paramètre kid dans le header accepte-t-il une injection de chemin ? Teste "kid":"/dev/null" avec l'algo HS256. Ou teste l'algorithme none.
Dans le panneau admin (/admin/templates), le champ "Nom du template" est rendu avec Jinja2. Teste {{7*7}}. Si 49 s'affiche, tu as une SSTI. Exploite avec {{lipsum.__globals__.os.popen('id').read()}}.
Depuis le reverse shell, scanne le réseau 10.0.0.0/24. Le serveur SMTP en 10.0.0.10 écoute sur le port 25. Utilise chisel pour créer un tunnel SOCKS et connecte-toi avec proxychains telnet 10.0.0.10 25.

5 Livrable attendu

Template de documentation ATT&CK à remplir

Pour chaque flag, documentez :

  • Technique MITRE : le ou les TXXXX correspondants
  • Étape de la kill chain : reconnaissance → initial access → execution → etc.
  • Payload utilisée : copier-coller la charge utile exacte
  • Impact : ce que la faille a permis d'obtenir
  • Remédiation recommandée : une phrase

Checklist

  • 🎯 Flag 1 — IDOR : profil d'un autre utilisateur lu
  • 🎯 Flag 2 — SQLi : table users extraite
  • 🎯 Flag 3 — Auth bypass : accès admin
  • 🎯 Flag 4 — SSTI → RCE : shell sur le serveur
  • 🎯 Flag 5 — Pivoting : fichier SMTP exfiltré
